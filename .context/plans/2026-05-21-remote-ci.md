# Sub-plan: BEAD-0016 — remote CI via GitHub Actions

Date: 2026-05-21
Branch: `ci/remote-github-actions`

## Problem

`master` branch protection requires a status check, but no remote CI exists to
produce it. Every PR sits at `mergeStateStatus: BLOCKED` and merges only via
`gh pr merge --admin`, bypassing all gates. See BEAD-0016.

## Goal

Add a GitHub Actions workflow whose job emits a status check named **`ci`**
(the check `master` branch protection expects per the bead's Resolution:
"add a GitHub Actions workflow that runs `cargo xtask ci` ... and name that
job as the required status check"). A green run satisfies protection; a red
run blocks the merge — `--admin` no longer needed.

## Design

One workflow: `.github/workflows/ci.yml`.

- **Triggers:** `pull_request` targeting `master`, and `push` to `master`.
- **Job id / name:** `ci` — the status check string branch protection requires.
  The workflow `name:` is also `CI` so the Checks UI reads cleanly.
- **Runner:** `ubuntu-latest` (CPU-only; matches bead note that e2e/pytest
  need a CPU-only path — no GPU runner available or required).
- **Toolchain provisioning** (mirrors `scripts/bootstrap.sh` prerequisites):
  - Rust: `dtolnay/rust-toolchain` pinned to `1.85.0` (workspace
    `rust-version`, edition 2024), with `rustfmt` + `clippy` components.
  - `uv`: `astral-sh/setup-uv`, with `python-version` read from
    `.python-version` (3.11). setup-uv provisions the pinned interpreter.
  - `lefthook`: installed via `uv tool install lefthook` because
    `scripts/bootstrap.sh` (run by `cargo xtask setup`) hard-`die`s if
    `lefthook` is not on PATH. `lefthook install` only writes git hook
    files — harmless in CI — but the binary must exist for setup to proceed.
- **Caching:**
  - `Swatinem/rust-cache` for the cargo registry + `target/` (keyed on
    Cargo.lock + job).
  - setup-uv built-in cache (`enable-cache: true`) for the uv download/wheel
    cache, keyed on `pyproject.toml` / `uv.lock`.
- **Steps:**
  1. `actions/checkout`
  2. install Rust toolchain
  3. install `uv` + Python
  4. rust-cache
  5. `uv tool install lefthook`
  6. `cargo xtask setup`  — venv, deps, maturin, workspace build
  7. `cargo xtask ci`     — fmt + clippy + ruff + mypy + cargo test + pytest + e2e
- **Permissions:** least privilege — `contents: read` only. No write scopes;
  no secrets consumed; no untrusted PR input interpolated into `run:` blocks.

## Divergence from `cargo xtask ci`

None. The workflow runs `cargo xtask ci` verbatim so local and remote gates
are byte-identical. The bead flags e2e as a risk ("confirm the e2e suite runs
in a GitHub runner"): the e2e scripts spawn the daemon/orchestrator/worker
locally over stdio JSON-RPC with no GPU or external service, so they run on a
standard `ubuntu-latest` runner. We keep e2e in the single `ci` job rather
than splitting it out — splitting would create a second check that branch
protection does not require and weaken the gate. If e2e proves flaky/slow on
the runner in practice, a follow-up bead can split it; for now keep parity.

## Pinned action versions (verified during review)

- `actions/checkout@v6` — current major.
- `dtolnay/rust-toolchain@1.85.0` — version-named ref (documented usage).
- `Swatinem/rust-cache@v2` — current major (v2.9.x).
- `astral-sh/setup-uv@v8` — current major. Note: this action has **no**
  `python-version-file` input; it takes `python-version` directly. We pass
  `"3.11"` explicitly with a comment that it must track `.python-version`.
  This only sets uv's default `UV_PYTHON`; `bootstrap.sh` re-reads
  `.python-version`, so the venv interpreter is pinned by the file regardless.
- `lefthook` is installed via `uv tool install lefthook` (PyPI package
  `lefthook` 2.x bundles the binary); `$HOME/.local/bin` is appended to
  `$GITHUB_PATH` so `cargo xtask setup` resolves it.

## Validation

- YAML parsed for well-formedness (python `yaml.safe_load`). The PyYAML
  `on:` → boolean-`True` quirk is cosmetic; GitHub Actions' own parser
  treats `on` as a mapping key correctly.
- Steps cross-checked against `xtask/src/main.rs` (`Ci` arm) and
  `scripts/bootstrap.sh` prerequisites.
- Cannot fully execute GitHub Actions locally; opening the PR triggers the
  workflow for the real run.
- Self code-review pass (no Task subagent available in this environment):
  least-privilege `permissions: contents: read`, no untrusted PR input
  interpolated into `run:` blocks, `pull_request` (not `pull_request_target`)
  trigger, every action pinned.

## Branch-protection follow-up (out of band, not code)

After this PR's first green run, the repo admin must set the required status
check on `master` protection to **`ci`** (the job name). That is a one-time
GitHub settings change, noted in the PR body — not something the workflow
file can do itself.
